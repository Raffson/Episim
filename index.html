Assignment 1


What was asked for this assignment?

For our first assignment we were asked to investigate the influance of r0 on the attack rate. The first parameter, r0, is a basic reproduction number, i.e. a measure for infectiousness of a disease. In other words, how many people will 1 person infect in a completely susceptible population. The second parameter, attack rate, represents the fraction of the population that has been infected at the end of the simulation.

What was delivered by EpiSim?

Due to the fact that newer versions of the stride simulator were pushed which would change the results dramatically, we decided to take a closer look at the different versions. Our findings are summarized in this pdf report, however it is in dutch. We may be planning on including an english version later, refer to this summary instead. Aside from that we have also bundled all our data, graphs & reports into a zip file.

What did we conclude?

After comparing the results & trying to interpret them, one thing can be said for certain. No matter what results we take, the attack rate increases if we increase r0. In the original results we would see that the number of infected people would be limited to some extent, i.e. about 20% of the population would be the maximum fraction of infected people assuming the worst case scenario. For the latest version this becomes practically the entire population for r0 values bigger than or equal to 4. At this point we decided to proceed and inspect the different versions where we noticed a few strange things. Some versions did not compile at first, but this was due to a recurring problem where 'sstream' was not included in RNManager.h which can be found in the folder "main/cpp/util". Another problem was a crashing simulator for which fixes where introduced during the second lecture (Tue. 13 Feb. 2018 @ 1300 UTC) but somehow those fixes managed to slip back out of the master branch during subsequent merges. Next, we noticed that some builds passed tests while other builds didn't. We figured out that this was due to different versions of BatchRuns.cpp which can be found in "test/cpp/gtester", where we assume that originally the absolute error was allowed to be 10% off. This changed to 50% by changing the factor 0.1 to 0.5, while in some other builds this factor was simply left out. As a results, the ASSERT_NEAR test would pass the tests if the factor was left out. As to which factor it should actually be, we are still trying to figure out that one ourselves. At last, we tried running and comparing the results for a number of different builds and tracked down a major difference in results to elisekaa's commit with hash "11510dc...". We believe that somewhere during the refractor of the "Vaccinator" class, something went wrong and as a result the immunization & vaccination profiles are not working properly. This also seems logical when one compares the results since it seems likely that an enitre population would get infected if the disease were highly contagious and nobody were immune or vaccinated. Looking at the original results, they also seem likely considering some people may be immune while most others would have been vaccinated.

Assignment 2


What was asked for this assignment?

The goal of this assignment was to establish reasonable bounds for the number of infected people. Therefore it is needed to run a large number of simulations with different seeds. Once bounds are established, we are to use the bounds to calculate a percentage relative to the mean and use this to adjust the regression tests within the project.

What was delivered by EpiSim?

EpiSim ran 5000 simulations with configuration file "run_default.xml" with logging level set to 'None' (for which we summarized our findings into this report) after making some adjustments to the simulator and writing a number of scripts. Specifically we added the ability to suppress the .csv files produced by the simulator in order to reduce the amount of data written during these 5000 simulations. We also introduced a small fix where SimulatorBuilder.cpp would still write to the log when the logging level is set to 'None'. Aside from that 3 scripts were written in python. The first script is seedTester.py, which (as the name suggests) will run simulations for a number of seeds that are read from a file. The second script is seedGenerator.py, which generates a number of unique seeds and writes them to a file. The last script, plotter.py (only in zip-file), is used to generate a histogram from the produced data and provide some basic statistical information like mean, standard deviation & an interval of 10% around the mean. To find out how to use these scripts, refer to the comments within the files or try using the '--help' option. At last, we tried changing the 'margin' in BatchRuns.cpp to 0.05 (=5%) instead of the original 0.1 (=10%). However this resulted into a failing test for 'test_tag: influenza_a' where the allowed difference simply became too small. This could be explained by the fact that all of our results are based on the default configuration which uses "disease_measles.xml". This implies we should also look at the results for other diseases. Since we realized this too late we had no more time left to investigate this properly. Again all our data, reports & graphs are bundled into the following zip file.

What did we conclude?

After trying out more values for 'margin' we conclude that it is possible to take a value as low as 3% (though 5% would be a more conservative value) while the tests would still pass, except for 'influenza_a' (as mentioned in the previous section) for which 8.8% would be the lowest possible 'margin' while still passing the tests. This could be the result of a bad seed in the sense that it coincidentally produces a result that's an outlier. In this case the 'target' value does not accurately represent the mean and should therefore be adjusted. Another possibility would be that the influenza disease simply produces different results, for example a bigger standard deviation would explain why a bigger 'margin' may be needed. Since our results for the default configuration (except for turning the logging level to 'None') are normally distributed, it is likely to be the same case with influenza. Pushing the first 100 seeds of 'random2.dat' for the influenza disease through the simulator with the same configuration used in the 'influenza_a' test case, we quickly confirm this suspicion regarding normal distribution of results. It also turns out that the mean is slightly different from the 'target' specified in BatchRuns.cpp for 'influenza_a'. However we must keep in mind that only 100 samples were used compared to the 5000 samples we originally used for establishing bounds on the default configuration, and thus this mean may still be a bad estimate. Even these 5000 results are still a fraction compared to the total of 4,294,967,295 (unsigned long) possible seeds. When comparing the standard deviation proportionally, i.e. devide std. deviation by mean (call this result 'X' for later reference), we see that for 'influenza_a' the fraction is larger compared to our original results. To determine whether this is caused by the influenza disease itself or by one of the other parameters in the configuration used by the test case, we would need to eliminate possible parameters that could cause the issue by running more simulations with different configurations. One possible explanation that comes to mind would be that r0 somehow affects the standard deviation, supposing that 'X' would decrease along with an increasing r0 value since our original configuration had a r0 value of 11 compared to 3 for the influenza configuration of the test case. But on the other hand the number of days to simulate is also different so it may just as well be this difference that causes the phenomenon. Thus we conclude that 5% would be a good value for 'margin', though before actually changing the 'margin' of our regression tests we would first need to properly confirm that the mean for 'influenza_a' is indeed off from the current 'target'. Furthermore we would need to investigate the influance of r0 on this ratio 'X', as well as looking at the influance of the number of simulation days. If these parameters turn out to be irrelevant, then we must dig further and see what the effects are of the seeding rate, immunity rate/profile & vaccination rate/profile. At last, a small statistical summary is given for the average results of the default configuration. For the results of the seperate .dat files refer to the report or 'stat-summary.txt' within the zip file.
